# Export-parity golden-тест (Б-2.3 / Б-2.4)

## Зачем

Страховка перед любым рефактором форматтеров экспорта (DOCX/MD/TXT), в первую
очередь — перед переходом на единый tree-walker. Один программный акт-эталон
(`fixture_act.py`) задействует все возможности модели данных; тест
(`test_export_parity.py`) проверяет, что каждый маркер фикстуры присутствует в
выводе каждого формата.

**Правило Б-2.3: рефактор форматтеров без зелёного golden-теста запрещён.**
Walker и любые правки `formatters/` начинаются с прогона этого пакета и
заканчиваются им же.

## Политика: presence, не равенство

Проверяется **полнота данных** (маркер есть в выводе), а НЕ байтовое равенство
и НЕ равенство подписей-ярлыков: подписи полей («Ответственный:» vs
«Ответственные:»), формат заголовков и плейсхолдеры различаются между
форматами **осознанно** (решение Д.3). Такие различия не ассертятся.

Известные потери данных зафиксированы `@pytest.mark.xfail(strict=True)`:
починка потери даст XPASS → тест упадёт → xfail снимается вместе с фиксом.
Список потерь — в reason'ах xfail-тестов.

Сверка нумерации (Б-2.4): literal-номеров в Word XML нет, тест симулирует
multilevel-счётчики Word по `numPr`/`ilvl` и сравнивает с `node.number`
структурных узлов `type='item'`.

Зеркало для превью — `tests/js/preview-golden-parity.test.mjs` (чистые
функции превью, без DOM).

## Как добавлять маркеры при новом типе блока

При добавлении нового типа блока (чек-лист — dev-guide §10.10 «Как добавить
новый тип блока конструктора»):

1. Добавь в `fixture_act.py` узел/запись словаря нового типа со строками-маркерами
   `GOLDEN_<TYPE>_<FIELD>` (строки уникальны во всей фикстуре).
2. Внеси маркеры в `MARKERS_ALL_FORMATS` (если данные обязаны быть во всех
   форматах) или заведи отдельную константу `MARKER_*` с проверкой в нужных
   форматах.
3. Если формат теряет данные нового типа — НЕ чини молча в этом пакете:
   зафиксируй `xfail(strict=True)` с причиной и реши судьбу потери отдельно.
4. Продублируй маркеры в JS-зеркале превью, если у типа есть чистые функции
   рендера.
