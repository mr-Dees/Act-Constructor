# Встраивание внешнего агента в Audit Workstation через iframe

Практическое руководство: как подключить к Audit Workstation (далее — **AW**)
отдельное приложение-агент так, чтобы оно появилось пунктом в боковом меню и
открывалось прямо внутри AW — со своим интерфейсом, но в общей рамке портала.

Документ самодостаточный: чтобы повторить интеграцию, других файлов читать не
нужно. Живой рабочий пример всего описанного ниже — домен `sqlagent`
(SQL-агент) в этом же репозитории; можно открыть его как образец.

---

## 1. Что это за способ и когда он подходит

Представьте, что AW — это бизнес-центр, а ваш агент — арендатор. Есть два
способа «заехать»:

- **Со своим отдельным входом** (этот документ). Агент остаётся самостоятельным
  приложением: свой процесс, свой код, свои зависимости. AW просто вешает на
  стене «окно» (iframe), через которое видно интерфейс агента. Быстро, ничего
  переписывать не надо, версии библиотек никому не мешают.
- **Полное вселение** — агент переезжает внутрь AW и становится его частью
  (общий процесс, общий интерфейс). Это отдельное руководство:
  [`agent-integration-inprocess.md`](agent-integration-inprocess.md).

**Способ с iframe подходит, когда:**

- у агента уже есть собственный веб-интерфейс, и переписывать его не хочется;
- у агента «тяжёлые» или конфликтующие зависимости (свой `torch`, `spark`,
  свои версии библиотек), которые проще держать в его собственном окружении;
- нужно подключить агента быстро и проверить «вживую», не затевая большую
  работу.

**Не подходит, когда** нужен единый сквозной интерфейс (общие шапки, диалоги,
тема оформления один в один) или общий доступ к данным внутри одного процесса —
тогда смотрите руководство по полному слиянию.

## 2. Картина целиком

```
                         браузер пользователя
                                  │
                                  ▼
        ┌─────────────────────────────────────────────┐
        │  Страница AW  /myagent                       │
        │  (шапка, боковое меню, рамка портала)        │
        │                                              │
        │     ┌────────────────────────────────────┐   │
        │     │  <iframe>  — «окно» в агента        │   │
        │     │                                     │   │
        │     │   интерфейс агента как есть         │   │
        │     │   (его HTML/JS/CSS)                 │   │
        │     └───────────────┬────────────────────┘   │
        └─────────────────────┼────────────────────────┘
                              │  всё внутри окна — это запросы
                              ▼  к ПРОЦЕССУ АГЕНТА, не к AW
        ┌─────────────────────────────────────────────┐
        │  Процесс агента (отдельный сервер на порту)  │
        │  свой backend + свой frontend + свои данные  │
        └─────────────────────────────────────────────┘
```

Ключевая мысль: **внутри окна работает сам агент.** AW даёт только рамку,
пункт меню, проверку доступа и адрес, по которому окно смотрит. Когда
пользователь что-то нажимает внутри окна — запросы уходят в процесс агента, AW
их даже не видит.

### Откуда берётся адрес окна

AW и агент в бою живут в одном изолированном контейнере на пользователя, поэтому
браузер достаёт агента «по соседству» через прокси-путь портала:

| Среда | Адрес, который AW подставляет в окно |
|-------|--------------------------------------|
| Локальная разработка | `http://localhost:<порт-агента>/` |
| Боевой контур (JupyterHub) | `/user/<пользователь>/proxy/<порт-агента>/` |

В боевом контуре окно и страница AW — **один и тот же origin** (один адрес),
поэтому браузер не считает агента «чужим сайтом» и не мешает встраиванию.

### Несколько принятых решений (просто чтобы вы понимали контекст)

- **Доступ в этом режиме защищён изоляцией контейнера.** Сам процесс агента
  слушает свой порт без отдельной авторизации — но достучаться до него можно
  только изнутри контейнера конкретного пользователя. Этого достаточно. Кто
  *видит пункт меню* в AW — регулируется ролью (см. шаг 6).
- **AW почти не меняется при доработке агента.** Поправили код агента —
  перезапустили его процесс, и окно показывает новое. Список мест, где правка
  AW всё-таки нужна, — в разделе 8.
- **Никаких постоянных соединений.** Окно — это обычная веб-страница агента;
  AW не держит с агентом сокетов и не пересылает данные туда-сюда.

## 3. Что нужно от самого агента (предпосылки)

Прежде чем трогать AW, убедитесь, что агент удовлетворяет трём условиям:

1. **Запускается отдельным процессом и слушает известный порт.** Например,
   `uvicorn app.main:app --port 8005`. Порт фиксируем — он понадобится AW.
2. **Имеет собственный веб-интерфейс**, который отдаётся по корню (`/`).
3. **Использует относительные пути к своему API.** Внутри интерфейса агента
   запросы должны идти как `fetch('api/...')` (без ведущего слэша), а не
   `fetch('/api/...')`. Это критично: под прокси-путём портала ведущий слэш
   увёл бы запрос мимо агента. Аналогия: относительный путь — это «спроси у
   соседней двери на этом же этаже», а `/api` — «спустись на первый этаж к
   главному входу», что под прокси ведёт не туда.

Если агент пока ходит по абсолютным путям — это единственное, что, возможно,
придётся поправить в *его* коде. Всё остальное ниже — на стороне AW.

## 4. Структура того, что добавляем в AW

AW собран из «доменов» — самостоятельных папок-модулей, которые сами заявляют
свои страницы и пункты меню. Интеграция агента — это новый маленький домен.
Заведём его по имени агента, например `myagent`:

```
app/domains/myagent/
├── __init__.py            # «паспорт» домена: имя, страница, пункт меню
├── settings.py            # настройки (порт агента, вкл/выкл)
└── routes/
    ├── __init__.py        # отдаёт список HTML-роутов
    └── portal.py          # сам роут страницы /myagent

templates/portal/myagent/
└── embed.html             # шаблон страницы: окно (iframe) или баннер

static/js/portal/myagent/
└── myagent-embed.js       # скрипт страницы (передаёт агенту тему/логин)

static/css/portal/myagent/
└── myagent.css            # стили окна и баннера
```

Дальше — по файлу за раз.

## 5. Пошаговая сборка

### Шаг 1. Настройки домена — `settings.py`

Заводим две настройки: порт агента и флаг «включён». Значения берутся из
переменных окружения с префиксом `MYAGENT__`.

```python
"""Настройки домена агента."""

from pydantic import BaseModel


class MyAgentSettings(BaseModel):
    """Настройки встраивания агента."""

    enabled: bool = True
    sidecar_port: int = 8005   # порт, на котором слушает процесс агента
```

### Шаг 2. «Паспорт» домена — `__init__.py`

Здесь домен сообщает AW: как меня зовут, какие у меня страницы, какой пункт
показать в меню. **Важная мелочь:** как только домен зарегистрирован, доступ к
его странице автоматически закрывается проверкой роли — отдельно её включать не
нужно (см. шаг 6).

```python
"""Домен агента: страница со встроенным через iframe интерфейсом."""

DOMAIN_NAME = "myagent"


def _build_domain():
    from app.core.domain import DomainDescriptor, NavItem
    from app.domains.myagent.routes import get_html_routers
    from app.domains.myagent.settings import MyAgentSettings

    return DomainDescriptor(
        name=DOMAIN_NAME,
        html_routers=get_html_routers(),
        settings_class=MyAgentSettings,
        nav_items=[
            NavItem(
                label="Мой агент",          # подпись в меню
                url="/myagent",             # адрес страницы
                icon_svg='<path d="..."/>', # иконка (содержимое <svg>)
                order=30,                   # порядок в группе меню
                active_page="myagent",      # для подсветки активного пункта
                group="Аналитика",          # раздел меню
                description="Краткое описание для подсказки",
            ),
        ],
    )
```

`routes/__init__.py` просто отдаёт список роутов:

```python
"""HTML-роуты домена агента."""

from app.domains.myagent.routes.portal import router as portal_router


def get_html_routers():
    return [portal_router]
```

### Шаг 3. Роут страницы — `routes/portal.py`

Роут делает три вещи: вычисляет адрес окна, проверяет, **жив ли процесс
агента**, и отдаёт шаблон. Проверка «жив ли» нужна, чтобы вместо страшной
браузерной заглушки «не удаётся установить соединение» показать аккуратный
баннер (см. шаг 4).

Проверка простая: пробуем за полсекунды открыть TCP-соединение к порту агента
на `127.0.0.1`. Получилось — агент жив; нет — показываем баннер. (Процесс агента
живёт в том же контейнере, поэтому проверяем именно `localhost`, независимо от
того, как до него идёт браузер.)

```python
"""Страница агента: окно (iframe) на его интерфейс либо баннер «недоступен»."""

import asyncio

from fastapi import APIRouter, Depends, Request
from fastapi.responses import HTMLResponse

from app.api.v1.deps.role_deps import get_user_roles
from app.api.v1.endpoints.auth import get_current_user_from_env
from app.core.config import get_settings
from app.core.navigation import get_nav_items_for_user
from app.core.settings_registry import get as get_domain_settings
from app.core.templating import get_templates
from app.domains.myagent.settings import MyAgentSettings

templates = get_templates()
router = APIRouter()


def _build_src(port: int) -> str:
    """Адрес, на который смотрит окно."""
    app_settings = get_settings()
    if app_settings.database.type == "greenplum":   # боевой контур
        user = get_current_user_from_env(truncate=False)
        return f"/user/{user}/proxy/{port}/"
    return f"http://localhost:{port}/"               # локальная разработка


async def _is_up(port: int, timeout: float = 0.5) -> bool:
    """Слушает ли процесс агента свой порт (быстрый TCP-connect)."""
    try:
        _, writer = await asyncio.wait_for(
            asyncio.open_connection("127.0.0.1", port), timeout=timeout
        )
    except (OSError, asyncio.TimeoutError):
        return False
    writer.close()
    try:
        await writer.wait_closed()
    except OSError:
        pass
    return True


@router.get("/myagent", response_class=HTMLResponse)
async def show_myagent(request: Request, roles: list[dict] = Depends(get_user_roles)):
    settings = get_domain_settings("myagent", MyAgentSettings)
    available = await _is_up(settings.sidecar_port)
    return templates.TemplateResponse(
        request,
        "portal/myagent/embed.html",
        {
            "active_page": "myagent",
            "topbar_title": "Мой агент",
            "nav_groups": get_nav_items_for_user(roles),
            "is_admin": any(r["name"] == "Админ" for r in roles),
            "myagent_available": available,
            "myagent_port": settings.sidecar_port,
            "myagent_src": _build_src(settings.sidecar_port),
        },
    )
```

### Шаг 4. Шаблон страницы — `templates/portal/myagent/embed.html`

Шаблон выбирает одну из двух картинок: окно (если агент жив) или баннер (если
нет). Баннер оформляем в стиле AW, а не оставляем браузерную ошибку.

```html
{% extends "portal/base_portal.html" %}

{% block title %}Мой агент{% endblock %}

{% block content %}
{% if myagent_available %}
<div class="myagent-embed">
    <iframe id="myagentFrame"
            src="{{ myagent_src }}"
            title="Мой агент"
            sandbox="allow-scripts allow-same-origin allow-forms allow-downloads allow-popups allow-modals"></iframe>
</div>
{% else %}
<div class="myagent-unavailable">
    <div class="myagent-unavailable-card">
        <h1 class="myagent-unavailable-title">Агент сейчас недоступен</h1>
        <p class="myagent-unavailable-message">
            Сервис не запущен или перезапускается. Запустите процесс агента
            на порту <code>{{ myagent_port }}</code> и обновите страницу.
        </p>
        <button type="button" id="myagentRetry" class="btn btn-primary">Обновить</button>
    </div>
</div>
{% endif %}
{% endblock %}

{% block init_script %}
<script type="module" nonce="{{ request.state.csp_nonce }}">
    import { AuthManager } from '{{ url_for('static', path='js/shared/auth.js') | versioned }}';
    import { PortalSidebar } from '{{ url_for('static', path='js/portal/portal-sidebar.js') | versioned }}';
    import { MyAgentEmbed } from '{{ url_for('static', path='js/portal/myagent/myagent-embed.js') | versioned }}';

    document.addEventListener('DOMContentLoaded', async () => {
        if (await AuthManager.requireAuth()) {
            PortalSidebar.init();
            MyAgentEmbed.init();
        }
    });
</script>
{% endblock %}
```

Про `sandbox` — это **самая важная строчка** для встроенного интерфейса, ей
посвящён раздел 8. Коротко: каждое слово в `sandbox` — это разрешение, которое
рамка выдаёт окну. Если чего-то не хватает — какая-то кнопка агента молча
перестаёт работать.

Про `nonce` — он обязателен: страницы AW защищены политикой безопасности (CSP),
и встроенные скрипты должны помечаться этим одноразовым токеном.

### Шаг 5. Скрипт страницы — `static/js/portal/myagent/myagent-embed.js`

Скрипт делает две вещи: оживляет кнопку «Обновить» на баннере и (по желанию)
передаёт встроенному интерфейсу логин пользователя и тему оформления.

```javascript
export const MyAgentEmbed = {
    init() {
        // Кнопка «Обновить» на баннере. Вешаем обработчик кодом, а не через
        // onclick в HTML — это запрещено политикой безопасности AW.
        const retry = document.getElementById('myagentRetry');
        if (retry) {
            retry.addEventListener('click', () => window.location.reload());
        }

        const frame = document.getElementById('myagentFrame');
        if (!frame) return;

        // Когда окно загрузится — передаём агенту логин и тему.
        frame.addEventListener('load', () => {
            let targetOrigin;
            try {
                targetOrigin = new URL(frame.src, window.location.href).origin;
            } catch (_) {
                return;
            }
            const user = (window.AuthManager && AuthManager.getCurrentUser)
                ? AuthManager.getCurrentUser()
                : null;
            frame.contentWindow.postMessage(
                { type: 'aw:init', user, theme: 'light' },
                targetOrigin,   // адресуем строго окну, не '*'
            );
        });
    },
};

// Дублируем на window — этого требует конвенция inline-скриптов портала AW.
window.MyAgentEmbed = MyAgentEmbed;
```

Передача логина/темы — задел на будущее: агента можно научить читать сообщение
`aw:init` и подстраиваться. Если этого не нужно — блок с `postMessage` можно
не добавлять, окно всё равно работает.

### Шаг 6. Стили — `static/css/portal/myagent/myagent.css`

Окно должно занять всю высоту, а баннер — оказаться по центру. Используйте
переменные оформления AW (`var(--...)`), чтобы баннер совпадал по стилю с
остальным приложением.

```css
.myagent-embed { flex: 1 1 auto; display: flex; flex-direction: column; }
.myagent-embed iframe { flex: 1; width: 100%; border: 0; }

.myagent-unavailable { flex: 1; display: flex; align-items: center; justify-content: center; }
.myagent-unavailable-card {
    max-width: 32rem; text-align: center;
    background: var(--card-bg); border: 1px solid var(--border-light);
    border-radius: var(--radius-xl); box-shadow: var(--shadow-xl);
    padding: 2rem;
}
.myagent-unavailable-title { font-size: var(--font-size-2xl); color: var(--text-primary); }
.myagent-unavailable-message { color: var(--text-secondary); }
.myagent-unavailable-message code { font-family: monospace; background: var(--bg-tertiary); }
```

И **обязательно** подключите этот файл в общий портальный набор —
`static/css/entry/portal.css`:

```css
/* Мой агент (встроенный iframe + баннер «недоступен») */
@import '../portal/myagent/myagent.css';
```

CSS в AW собирается без сборщика — без этой строки стили просто не загрузятся.

### Шаг 7. Доступ — роль в админ-панели

Доступ к разделам в AW раздаётся ролями. В админ-панели у каждого пользователя —
ряд «капсул» (по одной на роль): нажал — выдал, нажал ещё — отозвал. Чтобы у
вашего агента появилась такая капсула, нужно завести роль с именем агента в
домене.

Роли перечислены сидом в схеме админ-домена. Добавьте строку в **оба** файла
(PostgreSQL и Greenplum):

`app/domains/admin/migrations/postgresql/schema.sql`:
```sql
INSERT INTO {SCHEMA}.{PREFIX}roles (name, domain_name, description) VALUES
    ...
    ('Мой агент', 'myagent', 'Доступ к агенту')
ON CONFLICT DO NOTHING;
```

`app/domains/admin/migrations/greenplum/schema.sql` — там сид в форме `UNION ALL`:
```sql
    ...
    UNION ALL SELECT 'Мой агент', 'myagent', 'Доступ к агенту'
```

> ⚠️ **Важно для уже работающих баз.** AW при старте создаёт только
> отсутствующие таблицы и не перезаписывает существующие. Если база уже
> развёрнута (таблица ролей есть), новая строка сама не появится — её нужно
> добавить разово вручную. Готовый скрипт — в
> [`docs/migrations/add-agent-role.md`](../migrations/add-agent-role.md).

**Чтобы доступ выдавался всем по умолчанию** (как к Конструктору и Чату),
добавьте имя роли в список дефолтных — `DEFAULT_ROLE_NAMES` в
`app/api/v1/deps/role_deps.py`:

```python
DEFAULT_ROLE_NAMES: tuple[str, ...] = ("Цифровой акт", "Чат-ассистент", "Мой агент")
```

Этот список назначается каждому новому пользователю автоматически. Уже
существующим пользователям роль так не достанется — для них см. тот же
скрипт миграции (там есть блок «выдать всем разом»).

Администратору роль не нужна — он видит все разделы по умолчанию.

### Шаг 8. Конфигурация — `.env` и `.env.example`

Добавьте блок настроек агента. В `.env.example` — обязательно (у пользователей
свой `.env` не обновляется автоматически):

```ini
# === Домен: Мой агент ===
MYAGENT__ENABLED=true
MYAGENT__SIDECAR_PORT=8005
```

> ⚠️ Проверьте, что порт агента **не совпадает** с портом самого AW
> (`SERVER__PORT`). Если совпадёт — окно будет смотреть в само AW, получится
> «зеркало в зеркале».

## 6. Как теперь устроена разработка агента

Самый частый вопрос: «я поправил агента — нужно что-то менять в AW?»

**Почти всегда — нет.** Интерфейс и backend агента живут внутри его процесса.
Поправили — перезапустили процесс агента — обновили страницу в AW. Окно
подтянет новое само. AW про внутренности агента ничего не знает.

Правка AW нужна только если меняется один из четырёх «стыков» (раздел 8).

## 7. Проверка, что всё работает (чеклист приёмки)

- ☐ Процесс агента запущен и слушает свой порт.
- ☐ В боковом меню AW появился пункт агента (под нужной ролью).
- ☐ Клик по пункту открывает страницу, в окне — рабочий интерфейс агента.
- ☐ Если процесс агента остановить — вместо браузерной ошибки виден аккуратный
  баннер, кнопка «Обновить» работает.
- ☐ Внутренние действия агента (запросы, скачивания, диалоги-подтверждения)
  работают.
- ☐ В консоли браузера нет ошибок про CSP или X-Frame.
- ☐ Пользователь без роли агента пункт не видит и получает отказ на странице.

## 8. Где AW всё-таки приходится трогать (четыре «стыка»)

В режиме с окном AW и агент почти не пересекаются — но есть ровно четыре места,
которые живут на стороне AW и которые нужно поправить, если у агента меняются
требования к браузеру:

1. **Порт** — `MYAGENT__SIDECAR_PORT` должен совпадать с портом, на котором
   реально слушает процесс агента.

2. **Разрешения окна (`sandbox`)** — самый коварный стык. Окно работает в
   «песочнице»: браузер по умолчанию запрещает ему почти всё, а каждое слово в
   атрибуте `sandbox` — это выданное разрешение. Если интерфейс агента вдруг
   «молча не работает» в окне (а отдельным процессом работает) — почти всегда
   не хватает разрешения.

   **Живой пример.** В интерфейсе SQL-агента есть кнопка «Сброс сессии». Её код
   начинается с подтверждения: `if (!confirm('Сбросить?')) return;`. Браузер в
   песочнице **молча блокирует** `confirm()`/`alert()`/`prompt()`, если в
   `sandbox` нет разрешения `allow-modals`. Заблокированный `confirm()`
   возвращает «нет», условие срабатывает, и обработчик просто выходит — кнопка
   «не работает», причём **без единой ошибки в консоли**. Лечится добавлением
   `allow-modals` в `sandbox`.

   Поэтому в шаблоне (шаг 4) `sandbox` уже содержит разумный набор:
   `allow-scripts allow-same-origin allow-forms allow-downloads allow-popups allow-modals`.
   Правило: **добавляйте разрешения под реальные потребности интерфейса агента,
   а не «на всякий случай».** Лишние разрешения снижают защиту.

   Шпаргалка по частым разрешениям:

   | Разрешение | Зачем нужно |
   |------------|-------------|
   | `allow-scripts` | интерфейс на JS вообще выполняется |
   | `allow-same-origin` | интерфейс видит свой origin (нужно почти всегда) |
   | `allow-forms` | работают формы |
   | `allow-modals` | работают `confirm()` / `alert()` / `prompt()` |
   | `allow-downloads` | работает скачивание файлов |
   | `allow-popups` | интерфейс открывает новые вкладки/окна |

3. **Политика безопасности страницы (CSP / X-Frame)** — трогать нужно, только
   если интерфейс агента тянет ресурсы извне доверенного origin. В обычном
   случае правок не требуется.

4. **Роль доступа** — если поменяли имя роли или правила выдачи.

Всё остальное (HTML, JS, backend самого агента) — внутри процесса агента, AW не
касается.

## 9. Короткое резюме

1. Завести маленький домен (`__init__`, `settings`, роут, шаблон, JS, CSS).
2. В роуте — проверить, жив ли процесс агента, и отдать окно или баннер.
3. Подключить CSS в общий портальный набор.
4. Завести роль и (по желанию) добавить её в дефолтные.
5. Прописать порт в `.env` / `.env.example`.
6. Дальше дорабатывать агента отдельно — AW трогать только на четырёх стыках.

Рабочий образец, на который можно опираться один в один, — домен `sqlagent` в
этом репозитории.
