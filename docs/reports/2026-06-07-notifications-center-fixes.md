# Доработки центра уведомлений и шапки — отчёт

Дата: 2026-06-07. Ветка: `notifications-center` (PR в master — за владельцем).
Продолжение основной поставки (`docs/reports/2026-06-07-notifications-center.md`).

Четыре правки по обратной связи: каждая — своя под-ветка, влитая в `notifications-center`.

| # | Под-ветка | Суть |
|---|---|---|
| R1 | `notifications-readall-fix` | Фикс HTTP 500 на «Прочитать все». |
| R2 | `header-buttons-even-spacing` | Равномерные отступы кнопок шапки конструктора. |
| R3 | `portal-bell-styling` | Своя (светлая) стилистика дропдауна колокольчика на портале. |
| R4 | `notifications-acts-status-source` | Живые уведомления по актам с незакрытыми требованиями. |

## R1 — «Прочитать все» падало с 500

**Причина.** Во втором запросе `NotificationRepository.mark_all_read` (`INSERT ... SELECT`)
параметр `$1` употреблялся и в списке `SELECT n.id, $1, ...` (где PostgreSQL выводил
тип как `text`), и в сравнениях `recipient_user_id = $1` / `s.user_id = $1` (где тип
выводился как `varchar` из колонок `VARCHAR(50)`). Конфликт → `asyncpg
AmbiguousParameterError: для параметра $1 выведены несогласованные типы (text и
character varying)`.

**Почему не поймали раньше.** Это ошибка вывода типов реального PostgreSQL. Юнит-тесты
репозитория работают на mock-соединении (SQL не исполняется), поэтому баг проявлялся
только на живой БД.

**Фикс.** Явное приведение `$1::varchar` в списке SELECT — все три употребления
параметра становятся `varchar`. Тот же приём уже применяется в проекте
(`admin_repository.py`, `role_deps.py`) по той же причине; `::varchar` валиден на
Greenplum 6.x (PG 9.4). Добавлена регрессия в `test_notification_repository.py`
(`assert "$1::varchar" in insert_sql`).

## R2 — равномерные отступы кнопок шапки

**Причина неравномерности.** Ширина колонки каждой кнопки равнялась ширине её
**подписи**, а у индикатора сохранности была дополнительно прибита `min-width: 6.5rem`.
Из-за этого колонки «Сохранено» и «Уведомления» (самые широкие подписи) были шире
прочих, и кнопки в группах стояли на разном расстоянии.

**Решение.** Подпись выведена из расчёта ширины колонки: `.header-action-label { width: 0;
overflow: visible; }` (плюс уже были `white-space: nowrap; text-align: center`). Подпись
центрируется по центру кнопки и симметрично «вытекает» за края, не растягивая колонку.
Колонка = ширине кнопки (2.5rem) → все колонки одинаковы. Удалены `min-width` индикатора
и осиротевший токен `--header-action-column-min-width`.

**Зазор между подписями (важно).** При `width:0` ничто не мешает длинным кириллическим
подписям наложиться на соседние — это поймало адверсариальное ревью. Реальные ширины
подписей измерены в браузере (Segoe UI, 12px, semibold 600 — как у пользователя на
Windows): «Уведомления» 76.5px, «Просмотр» 57.1px, «Только локально» 96.1px (самое
широкое состояние индикатора), кнопка 40px. При прежнем `gap` 1.5rem (24px) пары
«Уведомления/Просмотр» и «Помощь/Только локально» **накладывались**. Зазор групп увеличен
до `--spacing-3xl` (2.5rem = 40px); измеренные просветы между ближайшими подписями стали
положительными во всех парах (минимум — «Уведомления/Просмотр» ≈ 5px, «Помощь/Только
локально» ≈ 33px). Раскладка проверена рендером в браузере (скриншот): кнопки равноудалены,
наложений нет, индикатор и колокольчик больше не выбиваются.

## R3 — портальная стилистика колокольчика

**Причина.** Шапка дропдауна использует общий `--gradient-header` (фиолетовый) — он
уместен в конструкторе (вся шапка фиолетовая), но на портале topbar светлый
(`.landing-topbar`: белый фон, серый текст), и фиолетовая шапка выглядела чужеродно.

**Решение.** В общий `notifications-center.css` добавлены оверрайды под предком
`.landing-topbar-actions`: светлая шапка (`--bg-primary` + `--text-primary` + нижняя
граница), перекрашенные «Прочитать все»/крестик (`--text-secondary`, hover `--gray-100`/
`--primary`), и меню чуть плотнее к topbar (`--spacing-sm`). Оверрайды изолированы:
колокольчик конструктора не находится внутри `.landing-topbar-actions`, поэтому там
остаётся градиент. Селекторы-потомки (специфичность 0,2,0) перекрывают базовые (0,1,0)
без `!important`.

## R4 — живые уведомления по актам

**Запрос владельца.** На портале было непонятно, какие уведомления приходят; в
конструкторе замечания по таблицам уместны. Нужно при загрузке актов проверять статусы
карточек (нужна фактура, СЗ и т.п.) и тоже показывать их в колокольчике.

**Решение — живой источник, не персист.** По образцу источника «tables» в конструкторе
добавлен живой источник «acts» (`static/js/portal/acts-manager/notifications-source-acts.js`):
акты с незакрытыми требованиями превращаются в уведомления колокольчика на странице списка
актов. Источник pull-based: страница хранит последний загруженный список (`_acts`) и дёргает
`refresh()`. Это снимок текущего состояния (как «tables»), он **не персистится** —
требования меняются по мере правки актов, и запись их в БД создавала бы устаревшие дубли.

**Маппинг.** Используются уже существующие флаги `ActListItem`:
`needs_invoice_check` → severity `error` (критично, как красная карточка), иначе `warning`;
`needs_created_date`/`needs_directive_number`/`needs_service_note` → перечисление в теле
(«Требуется: проверка фактуры, служебная записка, …»). Заблокированные и «готовые» акты
пропускаются. Клик ведёт к акту (`openAct` → `/constructor?act_id=…`, proxy-safe).
`inspection_name` (пользовательский) рендерится центром через `textContent` — XSS нет.

**Решение по персистентному продьюсеру актов (T3).** Оставлен без изменений. «Системные
события актов» (создание/экспорт) — отдельный, явно запрошенный источник; новый живой
источник его дополняет (разные пространства id, конфликта нет), закрывая именно нехватку
**actionable**-уведомлений на портале.

## Верификация

- Полный прогон: **pytest 1627 passed, node:test 184 passed** (новые 8 тестов чистого
  построителя «acts» + регрессия `$1::varchar`).
- Каждая правка прошла адверсариальное ревью (по одному скептику + охотник за латентными
  багами); единственная находка (наложение подписей в R2) исправлена и проверена замером.
- **R2 проверен визуально** в браузере (реальный Segoe UI Windows): замер просветов между
  подписями + скриншот раскладки.
- **Не проверено на живом приложении** (запущенный сервер был недоступен из среды): фактическое
  исполнение `read-all` против реального PostgreSQL (R1) и рендер портального дропдауна (R3)
  стоит глянуть на стенде. Бэкенд-эндпоинты должны быть задеплоены, иначе колокольчик отдаёт
  пустой снимок (не ломается).
